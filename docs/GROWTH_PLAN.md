## 1. ゴールと現状のギャップ

**ゴール**: まず10家族、その後30家族が、それぞれ自分の薬・便の状態区分・通知設定で使える状態。

**現状の構造的制約**(1家族専用に作られている):

| 項目 | 現状 | 30人対応に必要なこと |
|---|---|---|
| 認証 | 全員共通の1つのPIN | ユーザーごとの本人確認 |
| データ | `gutpacer-logs` のキーが日付のみ | ユーザーIDでの分離(他人の記録が混ざる/見える構造を根本から変える) |
| 薬 | モビコール朝昼夕の固定チェックボックス | 薬の名前・回数・タイミングを各自登録 |
| 便の状態 | 4種類の固定選択肢 | 選択肢の編集(ブリストルスケール準拠も選べるように) |
| 通知 | 環境変数の1人のLINE IDへ送信 | 各ユーザーの LINE へ、各自のしきい値で送信 |
| 通知しきい値 | 1日でリマインド・2日で警告(固定) | 人によって排便ペースが違うため設定可能に |

## 2. 技術方針(重要な決定3つ)

### 決定1: 認証は LINE ログイン(LIFF)にする

Cognito 等の汎用認証ではなく LINE を使う理由:

- 通知が既に LINE。ログインも LINE なら、**ユーザー登録 = 公式アカウントを友だち追加してアプリを開くだけ**になり、介護者層(50〜70代)にパスワード管理をさせずに済む。
- LINE ユーザーIDがそのまま通知先IDになるので、通知のマルチユーザー化と一石二鳥。
- 招待リンク(QRコード)を渡すだけでオンボーディングが完了し、医療関係者経由の紹介と相性が良い。

現行PINは開発・緊急用のバックドアとして残してもよいが、一般ユーザーには見せない。

### 決定2: DynamoDB はユーザーIDをパーティションキーにした新テーブルへ移行

- `gutpacer-logs-v2`: PK = `userId`, SK = `fullDate`
- `gutpacer-users`: PK = `userId`(プロファイル: 表示名、薬リスト、便選択肢、通知しきい値、location、LINE通知先)
- 既存1家族分のデータは移行スクリプトで `userId = <あなたのID>` として `-v2` にコピー。移行完了まで旧テーブルは消さない。
- Scan は使わない設計に改める(現行 GET は全件 Scan。ユーザー単位の Query に置き換え)。

### 決定3: カスタマイズは「プロファイル駆動フォーム」で実現

フォームの薬セクション・便セクションをハードコードせず、ユーザープロファイルの定義から描画する:

```json
{
  "meds": [
    { "id": "movicol", "name": "モビコール", "slots": ["朝", "昼", "夜"] },
    { "id": "magmitt", "name": "酸化マグネシウム", "slots": ["夜"] }
  ],
  "stoolTypes": ["硬い（コロコロ）", "普通（バナナ状）", "..."],
  "notify": { "remindAfterDays": 1, "warnAfterDays": 2 }
}
```

- 初期値は現行と同じ内容(移行ユーザーは何も変わらない)。
- 便の状態はプリセットとして「現行4種」と「ブリストルスケール7種」を用意し、自由編集も可能に。
- 記録データには選択時の**文字列を保存**する(定義を後から変えても過去の記録の意味が変わらないように)。

## 3. 開発タスク10(実行順)

### 基盤(〜1ヶ月)

- [ ] **G-1: LINE Login/LIFF 認証の導入** — LIFF アプリ化し、`X-Pin` の代わりに LIFF の ID トークンを API に送り Lambda 側で検証。※LINEチャネル作成はあなたの作業(USER_TODO 参照)
- [ ] **G-2: DynamoDB マルチテナント化** — 新テーブル作成(PITR 有効で)、API の全操作を userId スコープの Query に書き換え、既存データ移行スクリプト。
- [ ] **G-3: API デプロイ自動化 + 検証環境** — 手動デプロイのまま30人に配るのは事故のもと。GitHub Actions 化し、本番と別の検証用 Lambda/テーブルを用意。

### カスタマイズ(1〜2ヶ月)

- [ ] **G-4: ユーザープロファイル API** — プロファイルの取得/更新エンドポイントとバリデーション。
- [ ] **G-5: フォームの動的描画** — 薬・便セクションをプロファイルから生成。履歴表示・PDF も動的化。
- [ ] **G-6: 設定画面 + 初回オンボーディング** — 初回起動時に「薬の登録 → 通知設定」を2〜3画面で完了させるウィザード。**ここの出来が継続率を決める。**

### 通知・運用(2〜3ヶ月)

- [ ] **G-7: notifier のマルチユーザー化** — 全ユーザーを走査し、各自のしきい値で判定、各自の LINE に push。エラーは1人分失敗しても他に波及させない。
- [ ] **G-8: 退会・データ削除 + 規約表示** — 自分のデータを全削除できる機能と、プライバシーポリシー/利用規約ページ。健康情報を預かる以上、10人時点で必須。
- [ ] **G-9: 運用の見える化** — 週次の記録継続率(アクティブ記録者数)を確認できる管理用ビュー + CloudWatch アラーム(API/notifier のエラー通知)。

### 公開準備(3ヶ月〜)

- [ ] **G-10: PWA 化** — manifest + アイコンで「ホーム画面に追加」に対応。アプリストアの代替(後述)。

各タスクは完了時に [TASKS.md](TASKS.md) に実績として転記する。

## 4. ユーザーの増やし方(質問への回答)

### 結論

**知り合いの医療関係者から始めるのが正解。アプリストアは30人までは不要(PWAで代替)。SNSは補助。大企業営業は今はやらない。**

### 0 → 10人: 医療関係者経由の手渡し(推奨ルート)

- 知り合いの訪問看護師・ケアマネ・神経内科関係者に「排便記録で困っている在宅介護の家族」を紹介してもらう。1人の医療者は複数の該当家族を知っているので、紹介元は2〜3人で足りる。
- この段階は**全員あなたが直接オンボーディング**する(LINEで友だち追加→初期設定を一緒に)。手間だが、つまずきポイントの発見がそのまま G-6 の設計に還元される。
- 医療者経由の利点: ①信頼が最初からある ②「記録を医師に見せる」という核の用途が紹介文脈と一致 ③フィードバックの質が高い。
- 注意: 医療者に「診療の一部」として使わせない(責任分界が曖昧になる)。あくまで**家族の記録ツールを家族に紹介してもらう**建て付けを守る。

### 10 → 30人: 口コミ + コミュニティ + コンテンツ

- 最初の10家族の満足が最大の獲得チャネル。招待用QRコード/リンクを渡し「同じ悩みの家族がいたら」と言えるようにする(G-1のLINE化でこれが1タップになる)。
- パーキンソン病友の会・介護者の会・地域包括支援センターの掲示や紹介。
- note/X で「パーキンソン病介護と便秘管理」の実体験コンテンツを月1〜2本。検索流入は少数だが、来る人は全員が対象者。
- 広告は打たない(STRATEGY.md の方針どおり。CPAが合わない)。

### アプリストアに出すべきか → 30人までは No

- 本アプリはWebで完結しており、G-10のPWA化で「ホーム画面のアイコンから起動」は実現できる。
- ストアに出すコスト: Apple Developer 年間約1.5万円 + 審査対応 + 更新のたびの審査待ち + 健康系アプリへの審査要件。30人規模ではリターンがない。
- 「ストアにある」ことの信頼効果が効くのは不特定多数への配布段階(100人〜)。その時に改めて判断。

### 大企業への営業を狙うべきか → 今は No、明確に非推奨

理由:

1. **時期尚早**: 企業(製薬・介護大手・保険)への営業は商談サイクル半年〜1年以上。その間、個人ユーザー獲得が止まる。30人という目標に対して遠回り。
2. **要求水準**: 法人契約には情報セキュリティ監査・ISMS等の認証・医療情報ガイドライン(いわゆる3省2ガイドライン)対応が求められがちで、現体制では受けられない。
3. **証拠がまだない**: 企業が買うのは「継続率と改善実績のデータ」。それは30人×6ヶ月の運用からしか生まれない。**つまり30人プランの完遂こそが、将来の法人営業の準備そのもの。**

例外として現実的な中間ルート: 大企業ではなく**個別の訪問看護ステーションやケアマネ事業所**(数人〜数十人の事業所)に「担当利用者の家族に紹介してもらう」無償連携を打診するのは、10→30人の段階で有効。これは営業ではなく紹介チャネルづくり。

## 5. 30人時点のコスト見通し

| 項目 | 月額見込み | 備考 |
|---|---|---|
| AWS(Lambda/DynamoDB/S3/CloudFront) | ¥100〜500 | 30人でも無料枠内が大半 |
| LINE公式アカウント | ¥0 → **¥5,000** | 無料枠は月200通。30人×月10通で超過するため、通知が増えた時点でライトプラン(月5,000通)へ |
| ドメイン等 | 既存流用 | veai.jp 同居 |
| **合計** | **月 ¥5,500 前後** | 有料化するなら1家族 月¥300 × 30人 = ¥9,000 で回収可能な水準 |

開発工数: G-1〜G-10 で **20〜30人日**(AI併用・副業ペースで3〜5ヶ月)。STRATEGY.md 5.2 の Phase 3+4 見積もりと整合。

## 6. 中止・見直しの基準(先に決めておく)

- 10人に配って**1ヶ月後に週3日以上記録している家族が半分未満**なら、拡大を止めて G-6(オンボーディング)と入力体験を作り直す。
- 30人到達後3ヶ月の継続率が50%を超えたら、その実績データをもって次の段階(有料化・事業所連携の本格化・ストア展開)を検討する。

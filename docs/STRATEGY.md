## 1. Fable がこのアプリ開発で使うスキル

GutPacer は「サーバーレス + 静的フロント + LINE通知」という構成なので、必要なスキルは広く浅くではなく、以下に集中する。

### 1.1 現在のコードベースで既に使っているスキル

| 領域 | 具体的スキル | 該当箇所 |
|---|---|---|
| フロントエンド | 素のHTML/CSS/JS(フレームワークなし)、localStorage、CDNライブラリ(html2canvas / jspdf) | `frontend/index.html` |
| バックエンド | Node.js (ESM) Lambda、REST風API設計、CORS/プリフライト対応、PIN認証 | `backend/index.mjs` |
| データ | DynamoDB(単一テーブル・日付キー設計) | `gutpacer-logs`, `gutpacer-settings` |
| 通知 | LINE Messaging API(Flex Message)、JST日付計算、通知ロジック(1日リマインド/2日警告) | `backend/notifier/index.mjs` |
| インフラ/CI | S3 + CloudFront 静的配信、GitHub Actions デプロイ、EventBridge cron | `.github/workflows/*`, `scripts/deploy-notifier.sh` |

### 1.2 今後の開発で追加投入するスキル

- **テスト設計**: Lambda ハンドラのローカルスモークテスト(サンプルイベント駆動)。package.json すら無い現状からの最小テスト基盤づくり。
- **セキュリティ**: notes 欄の HTML エスケープ(XSS対策)、PIN のレート制限、秘密情報の環境変数管理。
- **IaC**: 手動デプロイされている API Lambda を CloudFormation/SAM/CDK のいずれかでコード化。
- **データ可視化**: 排便・モビコール服用の相関を医師に見せるためのグラフ/PDFレポート改善。
- **アクセシビリティ**: 高齢介護者向けの大きいタップ領域・高コントラスト・少ない入力ステップ。
- **医療系プロダクトの言葉選び**: 診断・治療助言に踏み込まない文言設計(handoff の方針を継続)。

### 1.3 意図的に使わない(過剰な)スキル

- React/Vue 等の SPA フレームワーク → 単一ページ・単一ユーザーの現状では複雑さが増すだけ。
- マイクロサービス分割、コンテナ → Lambda 2本で足りている。
- 独自認証基盤(Cognito等) → マルチユーザー化(Phase 4)まで不要。

## 2. ユーザーストーリー詳細検討

### 2.1 ペルソナ定義

| ペルソナ | 説明 | 主な接点 |
|---|---|---|
| **P1: 主介護者(家族)** | パーキンソン病の親を在宅介護する50代。毎日の記録者であり、このアプリの現在の唯一ユーザー。ITは日常レベル。 | スマホでアプリ入力、LINE受信 |
| **P2: 本人(患者)** | パーキンソン病当事者。便秘はパーキンソン病の代表的非運動症状で、モビコール(便秘薬)を定期服用。自分では記録しない/できない日がある。 | 間接的(記録される側) |
| **P3: ショートステイ等の施設スタッフ** | 本人が施設滞在中のケア担当。家族のアプリには触らないが、滞在中の排便情報のギャップが生まれる。 | 現状は接点なし(location=facility で通知停止のみ) |
| **P4: 主治医・神経内科医** | 外来で「最近の排便どうですか」と聞く。口頭のあいまいな回答より記録が欲しい。 | PDFエクスポートの受け手 |
| **P5: 副介護者(別居の兄弟等)** | 普段は関与しないが状況は知りたい。主介護者が倒れたときのバックアップ。 | 将来のマルチユーザー候補 |

### 2.2 実装済みストーリー(現状の棚卸し)

- ✅ P1として、毎日の排便(量・便形状・浣腸・摘便)とモビコール服用(朝昼夕)を1画面で記録したい。入力が面倒だと続かないから。
- ✅ P1として、排便が1日なければリマインド、2日以上なければ警告をLINEで受けたい。気づかないうちに便秘が進行すると腸閉塞リスクがあるから。
- ✅ P1として、本人が施設にいる間は通知を止めたい。記録できないのに警告が来ると通知が「オオカミ少年化」するから。
- ✅ P1として、受診時に記録をPDFで医師に見せたい。口頭説明では正確に伝わらないから。
- ✅ P1として、PINで保護したい。健康情報をURLを知っただけの他人に見られたくないから。

### 2.3 次に作るべきストーリー(優先度順・受け入れ条件つき)

#### S-1: 記録の安全性(優先度: 高)
> P1として、notes に何を書いてもアプリが壊れたり変な表示にならないでほしい。「<」や引用符を含むメモを書くことがあるから。

- 受け入れ条件: 履歴表示・PDF出力の両方で notes/自由入力がHTMLエスケープされる。`<script>` と入力しても文字として表示される。
- 背景: handoff の Known Gaps に記載済みのXSSリスク。ユーザーが家族1人でも、記録喪失は致命的。

#### S-2: 記録の見返し・傾向把握(優先度: 高)
> P1として、直近2〜4週間の排便間隔とモビコール量の関係をグラフで見たい。「薬を増やした週に効いたか」を医師と話すため。

- 受け入れ条件: 履歴画面に週単位のカレンダー/バーチャート表示。排便あり日・警告日・モビコール服用回数が一目でわかる。PDFにも同じビューが入る。
- 細かい検討: パーキンソン病外来は1〜3ヶ月間隔が普通なので、表示期間は「前回受診から今日まで」を選べると理想。まずは固定28日で十分。

#### S-3: 入力忘れリカバリ(優先度: 中)
> P1として、昨日の入力を忘れたときに過去日付の記録を簡単に追加・修正したい。忙しい日は夜に記録できないことがあるから。

- 受け入れ条件: 日付を選んで過去分を入力でき、通知ロジックが「記録なし」と「排便なしと記録」を区別する。
- 細かい検討: 現在の notifier は `bowel != null` で判定している。「記録忘れ」の日に警告LINEが来るのは実は正しい挙動(リマインドとして機能する)なので、壊さないこと。

#### S-4: 施設滞在の情報ギャップ解消(優先度: 中)
> P1として、施設から帰ってきたとき滞在中の排便有無をまとめて入力したい。施設の連絡帳には書いてあるが、アプリの記録が途切れると傾向が読めなくなるから。

- 受け入れ条件: location=facility 期間を履歴上「施設」と表示し、帰宅後に一括入力できる。
- 細かい検討: 施設スタッフ(P3)に直接入力させる案は、アカウント管理・責任分界の問題が大きく Phase 4 送り。まず「後から家族が転記」で解決する。

#### S-5: 副介護者への共有(優先度: 低→製品化時に高)
> P5として、自分は記録しないが姉が付けている記録と警告を自分のLINEでも受けたい。姉に何かあったとき状況が全くわからないのは困るから。

- 受け入れ条件: LINE通知の宛先を複数登録できる(閲覧専用リンクは次段階)。
- 細かい検討: 現状 `LINE_USER_ID` は単一の環境変数。複数宛先は settings テーブルに移すだけで実現でき、マルチユーザー化の最初の一歩として手頃。

#### S-6: 服薬調整の記録(優先度: 低)
> P1として、モビコールの量を変えた日(1包→2包など)を記録したい。医師の指示変更と効果の対応を後から追えるようにするため。

- 細かい検討: 「診断・治療助言をしない」方針は維持。アプリは事実の記録のみ行い、増減の提案は絶対にしない。

### 2.4 やらないと決めるストーリー

- ❌ 食事内容の記録 → 記録負担が跳ね上がり、核である排便記録の継続率を下げる。
- ❌ AIによる「便秘改善アドバイス」→ 医療助言リスク。handoff の方針どおり回避。
- ❌ 汎用健康記録アプリ化 → 「パーキンソン病の排便管理」という尖りが最大の資産。

## 3. 開発戦略

### Phase 1: 足場固め(〜1ヶ月)— 「壊れない・失わない」

今は動いているが、テストゼロ・手動デプロイ混在・レガシー併存という状態。機能追加の前にここを締める。

1. `backend/index.js`(レガシー)を削除または `legacy/` へ隔離し、`index.mjs` を唯一の実装と明記。
2. `package.json` + Lambda ハンドラのスモークテスト(サンプルイベントを流すだけの node スクリプト)。
3. notes 等のHTMLエスケープ(S-1)。
4. API Lambda のデプロイ手順の文書化 or GitHub Actions 化(main ブランチ運用の確定)。
5. DynamoDB のオンデマンドバックアップ or PITR 有効化。**記録データは再取得不可能**なのでこれが実質最優先。

### Phase 2: 記録体験の完成(1〜3ヶ月)— 「続けられる・役に立つ」

- S-2 グラフ/傾向表示 + PDFレポート強化(受診で「使える」体験を完成させる)
- S-3 過去日付入力、S-4 施設期間の一括入力
- 表示の高速化・オフライン耐性(直近データの localStorage キャッシュ)

### Phase 3: 一人→数人(3〜6ヶ月)— 「家族チーム対応」

- S-5 LINE通知の複数宛先
- 閲覧専用の共有ビュー(PINとは別の読み取り専用トークン)
- この段階までは**シングルテナントのまま**。自分たちの介護に最適化し続ける。

### Phase 4: 製品化判断(6ヶ月〜)— 「他の家族にも配れるか」

ここで初めてマルチテナント化(Cognito等の認証、テナント分離、LINE公式アカウント運用)を検討する。判断基準:

- Phase 2〜3 の機能で自分たちの運用が6ヶ月安定継続しているか
- 後述のマーケティング検証で「使いたい」という介護家族が10組見つかるか

見つからなければ、OSS + 構築手順公開(セルフホスト型)という出口も十分に価値がある。

### 開発運用の原則

- ブランチ: `development` で開発 → `main` マージでデプロイ、を正とする(現状の workflows と整合)。
- `frontend/config.js` は今後も git 管理外。環境依存値は同様に外出しする。
- 依存は最小に保つ。CDN 2ライブラリ + AWS SDK のみという軽さは、個人介護プロジェクトの持続可能性そのもの。

## 4. マーケティング戦略

### 4.1 前提の整理

GutPacer の本質は「パーキンソン病介護における便秘管理」という**極めて狭く深いニッチ**。便秘はパーキンソン病患者の6〜8割が経験する代表的非運動症状で、放置すると腸閉塞・入院につながる。一方で「排便記録アプリ」は世の中に多数あるが、**介護者が記録し・LINEで介護者に警告が飛ぶ**設計のものはほぼない。ここが差別化点。

### 4.2 段階別戦略

#### 段階A: 検証(Phase 2〜3 と並行)— お金をかけない

- **発信**: 開発と介護の実体験を note/ブログ/X で記事化。「パーキンソン病 便秘 記録」「介護 排便 アプリ」は検索ボリュームは小さいが競合コンテンツもほぼ無く、少数の深い読者に届く。
- **コミュニティ**: 全国パーキンソン病友の会、介護者の会、X上の介護アカウント界隈で「こういうものを作った」と共有し、10組のテスト家族を探す。
- **測るもの**: ①記録の継続率(週5日以上入力が続くか) ②LINE警告への反応(警告後に対処行動が起きたか)。DL数ではなく継続率が全て。

#### 段階B: 配布(製品化判断後)

選択肢は3つ。推奨は 1 → 2 の順で試すこと。

1. **OSS/セルフホスト公開**: GitHub公開 + 構築手順。技術のある家族・エンジニアの介護者に届く。信頼獲得とフィードバック源として最良。収益ゼロだが維持コストもゼロ。
2. **ホスティッド版(小規模有料)**: 月額300〜500円 or 買い切り。LINE公式アカウント経由で通知。数十〜数百家族規模を想定した「小商い」設計。
3. **B2B2C(訪問看護ステーション・ケアマネ経由)**: 排便記録は訪問看護の標準業務であり、家族との情報共有ニーズが実在する。ただし営業・契約・個人情報管理の負荷が跳ね上がるため、段階Bで手応えが出てから。

#### 避けること

- 広告出稿: ニッチすぎてCPAが合わない。検索・コミュニティ・口コミで十分。
- 「医療機器」的な訴求: 「便秘を改善」「健康になる」等の効能表現は薬機法リスク。**「記録と共有を楽にする」だけを言う**。
- 大手ヘルスケアアプリとの機能競争: 汎用化した瞬間に負ける。

### 4.3 メッセージング(たたき台)

> 「介護している家族のための、排便とお薬の記録アプリ。2日出ていなければ、LINEが教えてくれる。」

主語を患者ではなく**介護者**に置くこと。これが既存アプリとの最大の違いであり、届けるべき相手も介護者だから。

## 5. 開発コスト見積もり

### 5.1 ランニングコスト(現状構成・月額)

| 項目 | 見積もり | 備考 |
|---|---|---|
| Lambda(API + notifier) | ¥0 | 無料枠内(月100万リクエスト)。1家族なら誤差以下 |
| DynamoDB | ¥0〜数十円 | オンデマンド。1日数レコードなら実質無料。PITR有効化で +数円 |
| S3 + CloudFront | ¥0〜100円 | 静的ファイルのみ。既存の veai.jp 配信に同居 |
| LINE Messaging API | ¥0 | 無料枠(月200通)で十分。1日1通×31日でも余裕 |
| Route53 / ドメイン | 既存流用 | veai.jp に同居のため追加なし |
| **合計** | **月 ¥0〜200 程度** | 数百家族規模になっても月数千円レンジ |

### 5.2 開発工数見積もり(Phase 別)

前提: Claude Code 等のAI併用個人開発。「人日」は実作業日(1日2〜3時間の副業ペースなら暦日はその2〜3倍)。

| Phase | 内容 | AI併用 実工数 | 従来型(参考) |
|---|---|---|---|
| Phase 1 | レガシー整理・テスト基盤・エスケープ・バックアップ・デプロイ整備 | 3〜5人日 | 8〜12人日 |
| Phase 2 | グラフ/傾向表示・PDF強化・過去日付/施設一括入力 | 5〜8人日 | 15〜25人日 |
| Phase 3 | 複数LINE宛先・閲覧専用共有 | 3〜5人日 | 10〜15人日 |
| Phase 4 | マルチテナント化(認証・分離・LINE公式アカウント) | 15〜25人日 | 40〜60人日 |
| **Phase 1〜3 計** | 自分たち用として完成 | **11〜18人日** | 33〜52人日 |

### 5.3 金額換算

| シナリオ | 金額 | 内訳 |
|---|---|---|
| **自力 + AI(現在の体制)** | **月 ¥3,000〜¥6,000 程度 × 開発期間** | Claude Code サブスク + AWS実費のみ。Phase 1〜3 で総額 ¥2〜4万円 |
| フリーランスに外注(Phase 1〜3) | ¥150〜300万円 | 33〜52人日 × ¥4.5〜6万円/人日 |
| 開発会社に外注(Phase 1〜3) | ¥400〜800万円 | 要件定義・PM・保守込み。この規模の案件を受ける会社は実際には少ない |
| Phase 4(製品化)を外注追加 | +¥250〜500万円 | 認証基盤・マルチテナント・運用設計 |

**結論**: このプロジェクトの経済合理性は「自力 + AI」一択。外注した場合の見積額(数百万円)に対し、実費は月数千円で済む。逆に言えば、製品化(Phase 4)に進む場合も外部資金は不要で、必要なのは資金ではなく**10組のテスト家族と6ヶ月の運用実績**である。

### 5.4 見積もりの前提と変動要因

- 工数は現行コードベース(HTML 660行 + Lambda 2本)の規模感を基準にした概算。
- Phase 4 は LINE公式アカウント(コミュニケーションプラン無料枠→ライトプラン月¥5,000)の費用が加わる可能性あり。
- 医療情報を預かるホスティッド版では、プライバシーポリシー整備・個人情報保護対応の(金額よりも)手間を見込むこと。

# タスク委譲ワークフロー(立案=上位モデル / 実装=builder)

このスキルは「戦略・計画はメインセッション(Fable等の上位モデル)、実装作業は builder エージェント(既定 Sonnet)」という役割分担を実行する。

## 手順

### 1. タスクの特定と計画(メインセッションの仕事)

- 引数がタスクID(G-n / T-n)なら `docs/GROWTH_PLAN.md` / `docs/TASKS.md` から内容を特定する。
- 引数が自由記述なら、まず既存タスクとの重複・矛盾を確認する。
- 実装に入る前に**メインセッションが**次を決める(ここは委譲しない):
  - 変更対象ファイルと変更方針
  - やらないこと(スコープ境界)
  - 完了の判定基準
- 仕様に曖昧さが残る場合はユーザーに確認してから進む。

### 2. モデルの選定

| タスクの性質 | model 指定 |
|---|---|
| 通常の実装・修正・テスト追加(既存パターンの踏襲) | `sonnet`(既定。指定不要) |
| アーキテクチャ変更を伴う実装(例: G-1 LIFF認証、G-2 テーブル移行)、複数ファイル横断の大改修 | `opus` |

### 3. builder への委譲

Agent ツールで `subagent_type: "builder"` を指定して起動する。プロンプトには必ず含めること:

- タスクIDと目的(1〜2文)
- 変更対象ファイルと変更方針(手順1で決めたもの)
- スコープ境界(やらないこと)
- 完了判定基準(npm test 全パス + タスク固有の条件)

複数タスクを並行させる場合は、同じファイルを触るタスクを同時に走らせないこと。

### 4. 検収(メインセッションの仕事)

builder の報告を受けたら:

1. `git diff` で変更内容を自分の目で確認する(報告の鵜呑みにしない)。
2. `npm test` を再実行する。
3. スコープ外の変更が紛れていないか確認する。
4. 問題があれば **SendMessage で同じ builder に修正指示**を出す(新規エージェントを立てない)。
5. 合格したら `docs/TASKS.md` に完了実績を転記し、ユーザーに要約報告する。

## 原則

- 戦略・優先順位・スコープの判断を builder に委ねない。builder から判断保留が返ってきたら、メインセッションが決めてから再指示する。
- デプロイとコミットの実行判断はユーザーに確認する(builder にも許可しない)。
